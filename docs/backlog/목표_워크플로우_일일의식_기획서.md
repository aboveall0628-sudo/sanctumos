1. 워크플로우 완전 부재 — 목표↔도트 사이 다리가 없음
2. 5년/10년 리포트 미구현

## 4. 합의된 정책

### 4.1 5년·10년 목표의 역할
**도트 직결** — 10년이 5년의 부모, 5년이 연간의 부모... 풀어가면 도트로. 단 중간 수정·전면 개편 가능성을 데이터 모델이 흡수해야 함.

### 4.2 워크플로우 입력
**합작** — 사용자 초안 + AI 제안/보완. 의사결정 시스템과 동일 철학.

### 4.3 워크플로우 스텝 형태
**체크리스트 + 시간 박스 결합** — 순서 있음, 각 스텝마다 예상 도트 수.

### 4.4 도트 분배 메커니즘
**자동화 없음** — 매일 묵상 후 사용자가 직접 선택. 워크플로우는 "가능성의 지도"일 뿐.

**이유**:
- 실제 작업 속도가 매번 다름 (확인 필요)
- 아웃소싱/조력자 가능성에 따라 도트 속도 달라짐
- 자동 예약은 사용자의 분별 기회를 빼앗음

### 4.5 목표 수정의 4가지 패턴
| 패턴 | 처리 |
|---|---|
| 표현 수정 | 본인 한 줄, cascade 없음 |
| 부분 수정 | 하위 분기/주간 일부 갱신 |
| 전면 개편 | 5년/연간/분기/주간 거의 전면 재구성 + 개편 회고 리포트 |
| 폐기 | 하위 트리 archived 또는 완전 제거 |

### 4.6 목표 수정과 의사결정의 연계
**모든 목표 수정은 의사결정의 결과**. 별도 진입점 없음. 도피 방지의 본질.

- 사용자가 "목표 바꾸고 싶다" → 의사결정 시스템 진입 → 소크라테스 5개 질문 → 결정 → 판례 자동 생성 → 판례가 목표 수정 트리거
- 자유 수정 X, 강제 회수 제한 X, 의사결정 통과 게이트만

### 4.7 개편 회고 리포트
**자동 생성**. 목표 개편 시점에 다음을 포함:
- 이전 목표 스냅샷 + 새 목표
- AI가 추론한 구→신 매핑 (carried_over / transformed / abandoned / new)
- 사용자가 신 워크플로우의 시작 진척도 확정
- 트리거 판례 참조

## 5. 새 데이터 모델

### 5.1 Workflow (신규)
```
Workflow {
  id
  parentGoalId             어떤 목표의 워크플로우
  title
  steps: [
    {
      id,
      order,
      title,
      estimatedDots,        예상 도트 수
      executor              "self" | "helper" | "external"
      status                "pending" | "in_progress" | "done" | "abandoned"
      linkedDotIds          실제 분배된 도트 (역참조)
    }
  ]
  generatedByDecision      합작 시 AI 초안의 출처 판례
  createdAt
  revisionLog              워크플로우도 수정 로그 (의사결정 동일 패턴)
}
```

### 5.2 RevisionReport (신규)
```
RevisionReport {
  id
  scope                    "10y" | "5y" | "yearly" | "quarterly" | ...
  triggeredByPrecedentId   어떤 의사결정에서 비롯됐는가
  oldGoalSnapshot          개편 전 목표 트리
  newGoal                  새 목표
  aiInferredMapping: [
    {
      oldSubGoalId,
      newSubGoalId,
      relationType: "carried_over" | "transformed" | "abandoned" | "new"
      confidence: 0~100,
      userConfirmed: boolean
    }
  ]
  oldWorkflowProgress      개편 전 진척도 스냅샷
  inheritedProgress        사용자가 인정한 신 시작 진척도
  userInheritanceDecision  계승/폐기 스텝 사용자 확정
  createdAt
}
```

### 5.3 도트 보강
- 기존 `linkedGoalId` 유지
- 추가: `linkedWorkflowStepId` — 어느 워크플로우 스텝에서 분배됐는가
- 추가: `executedBy` — "self" | "helper" | "external"
- 추가: `goalAtTime` 스냅샷 (시점 보존, 압축 형태 검토)

## 8. v2 추가 합의 — 묶음 A/B/C 디테일

### 묶음 A — 데이터 모델
| 항목 | 결정 |
|---|---|
| **executor** | `self` / `helper`(인물 카드 참조) / `external` 3구분 |
| **시점 스냅샷 (R1)** | **방법 A** — 목표는 별도 versions 컬렉션에 v1/v2/v3 보존, 도트는 `goalVersionId`만 참조 |
| **고아 도트 (R5)** | 목표 폐기 시 **도트 삭제** + **개편 회고 리포트에 추모비**로 기록 |
| **추모비 내용** | ① 도트 수 + 실행 분포(완료/부분/건너뜀/대체) ② 기간(시작→폐기, 일수) ③ 만족도 평균 + 주요 레이블 통계 ④ 대표 도트 발췌 3개(메모 특이한 것 위주, AI 선별) ⑤ **contributions** — 사람×능력×기여도 (B-4 트랙에서 본격) ⑥ AI 내러티브 요약 |

### 묶음 B — AI 보조
| 항목 | 결정 |
|---|---|
| **구→신 매핑 UX** | 신뢰도 % 표시 + **각 매핑 항목마다 사용자가 승인/거부/수정** |
| **수정 등급 판정** | **AI 분석 제안 + 사용자 확인/변경** (표현/부분/전면/폐기 4단계) |

### 묶음 C — UI/Reports
| 항목 | 결정 |
|---|---|
| **일일 의식 — 웹** | **좌-우 2패널** — 좌: 워크플로우 스텝 목록 / 우: 내일 시간표 (드래그 앤 드롭) |
| **일일 의식 — 앱** | **2탭 스와이프** — 스텝 탭 ↔ 시간표 탭. 스텝에서 [+] 누르면 시간표 탭으로 자동 이동 |
| **5y/10y 리포트** | **AI 질의응답 + 본인 프로필 base** (별도 트랙 — 아래 §10) |
| **워크플로우↔Reports 연결** | 각 계층 리포트에 **'활성 워크플로우 진척' 섹션** + 별도 워크플로우 **상세 페이지** 둘 다 |

## 9. 새 데이터 모델 보강 (v2)

### 9.1 도트 보강 (v2)
- 기존 `linkedGoalId` 유지
- 추가: `goalVersionId` — 시점 스냅샷 참조 (방법 A)
- 추가: `linkedWorkflowStepId` — 어느 워크플로우 스텝에서 분배됐는가
- 추가: `executor` — "self" | "helper" | "external"
- 추가: `helperPersonId` — executor=helper일 때 인물 카드 ID

### 9.2 GoalVersion (신규)
```
GoalVersion {
  id
  goalId             원본 목표 ID
  versionNumber      v1, v2, v3...
  snapshotData       그 시점 목표 전체 (title/body/strength/...)
  validFrom          이 버전이 적용 시작된 시점
  validTo            다음 버전이 등장한 시점 (또는 null=현재)
  revisionReason     이 버전이 만들어진 이유 (revisionLog에서 인용)
}
```
**원칙**: 도트는 자기 시점의 `goalVersionId`만 참조. 컨텍스트 복원 시 `goalId + goalVersionId`로 GoalVersion 조회.

### 9.3 ExtinguishedGoalMemorial (추모비, 신규)
```
ExtinguishedGoalMemorial {
  id
  goalSnapshot          폐기 시점의 마지막 목표 스냅샷
  duration              { startDate, endDate, daysElapsed }
  dotStats              { total, completed, partial, skipped, replaced,
                         averageExecutionSatisfaction,
                         averageOutcomeSatisfaction,
                         topLabels }
  representativeDots: [ AI 선별 3개
    { dotSnippet, reasonChosen }
  ]
  contributions: [      ← B-4 트랙에서 본격 설계
    { personId, dotCount, skillsObserved, qualityScore, memorableMoments }
  ]
  aiNarrativeSummary    AI가 쓱 읽기 좋게 한 문단 요약
  triggeredByPrecedentId 어떤 의사결정으로 폐기됐는가
  createdAt
}
```

## 10. 본인 프로필(Self Profile) 트랙 — 별도 분리

> "AI가 질문하고 사람이 답하는 형식으로 부트스트랩하고,
> 도트 데이터에 따라 프로필 자동 갱신"

### 핵심 정책 (이번 세션 박은 것)
- 인물 카드와 **동일 구조**, 단 대상이 자기 자신
- 데이터 없을 때 **AI 질의응답 부트스트랩** (5y/10y 리포트의 base case)
- 도트 누적 시 자동 갱신 — 단 "**평가보다 인과**" 원칙 (자동 점수화 X, 관찰된 행동 사실만 누적)
- 5y/10y 리포트는 본인 프로필 데이터 + 누적 도트 + AI 질의를 결합해 narrative 생성

### 본격 설계는 B-4 통합 트랙
- 본인 프로필
- 인물 카드 점수 정책 (이미 메모리에 있음)
- 관계 신뢰도 (B-4)
- 양치기 소년 장치 (B-5)
- 추모비 contributions 차원
- → 한 트랙으로 묶어 별도 세션

## 11. v2 추가 — 정직성 인프라 (필드별 source 메타)

자기합리화 방지의 기반으로 **모든 데이터 모델에 필드별 source 메타** 박기.

```
도트/판례/원칙/메모 모든 필드에:
source: "self_report" | "calendar_sync" | "device" | "helper_input" | "ai_inferred"
```

- 객관 데이터(외부 검증 가능)와 주관 진술 자동 구분
- AI 응답·리포트에서 출처별 처리 가능
- 본격 자기합리화 방지 시스템은 B-4 통합 트랙에서 (B-6으로 분류)
