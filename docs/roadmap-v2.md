## 3. Spiritual Lock — 4 Layer → 1 Layer 로 축소

### 외부 문서의 야심
- Layer 1 (Daily Gate) — 토큰 없으면 `/today/word` 강제 라우팅
- Layer 2 (Mini Gate) — 타임박스 추가 / stance 부정 변경 / 100만원+ 거래 등 30~60초 게이트
- Layer 3 (Time Reservation) — 06:30 · 22:00 자동 슬롯 드래그 차단
- Layer 4 (Mandatory Alarm) — Service Worker Web Push 점진적 알람

### 현실 (2026-05)
실제로 작동하는 건 **자동 잠금 (15분 무활동)** 하나. 사실상 Layer 4 의 약한 버전.

### 자가 모순 정리
외부 문서는 동시에 "Grace Layer (Sabbath / Recovery)" 를 강조한다. 그런데:
- Layer 1 = 강제 라우팅
- Layer 2 = Skip 불가
- Layer 4 = 필수 알람

"율법주의 회피" 와 "강제 게이트" 가 정면 충돌. 사용자가 결국 시스템을 "감옥" 으로 느낄 위험.

### 결정
- **Layer 1~3 미구현 유지** (재설계 후 검토)
- 재설계 원칙: **강제 < 안내**. 모든 게이트는 Skip 가능 + 토글 가능
- "30초 기도 강제" 같은 시간 제한 모두 옵션화

## 4. 공동체 모듈 — 별도 트랙으로 분리

외부 문서의 §1.2.11 (공동체 모듈) · §1.2.12 (합류 훅 8종) · §2.1.3 (팀 키 매니저) · §4.2.8 (공동체 시나리오) 은 **Sanctum OS v2 에 포함하지 않는다**.

이유:
- **본질적으로 다른 시스템**: 단일 사용자 영성 도구 vs 다중 사용자 협업 도구. 보안 모델·DB·UX 다 다름
- **카카오 비즈채널 = 사업자 등록 = 법인/개사** 필요. 1인 영성 도구 사용자가 알림톡 월 결제 부담 불합리
- **RSA 팀 키 매니저** 1인 개발자가 직접 구현하기 위험 — 키 분실 시나리오, 멤버 추방 시 데이터 회수 본질적 불가
- 1인 개발 자원으로는 풀스코프 v3 = 5년 작업

**현재 결정**: 공동체 모듈은 별도 SaaS 프로젝트로 분리 검토. Sanctum OS 본체는 단일 사용자 영성 도구로 유지.

(외부 문서를 별도 파일 `docs/community-future.md` 로 옮기고 싶으면 그때 결정)

## 5. 카카오 알림톡 — Phase F 이후 옵션

**기본 채널**: Web Push (Service Worker). 이미 `ui/app.js` 에 SW 등록 코드 있음.

**알림톡**: 공동체 모듈과 함께 미래 옵션. 사업자 등록 + 비즈채널 심사 (2~4주) + 7종 템플릿 사전 승인 필요. 발송 실패율 5% 초과 시 Web Push fallback.

## 6. AI 비용 모델

현재 `llmProxy` Cloud Function 미배포 → 모든 AI 호출이 `ui/aiClient.js` 의 로컬 fallback 으로 처리됨 (템플릿 텍스트).

**Phase F 배포 시 비용 정책**:
- 개인 단일 사용자 자기 부담 (사용자 본인 Gemini API 키 입력 옵션)
- 또는 Cloud Functions 무료 한도 내 (월 200만 호출). 단일 사용자라 한도 안 넘김
- 토스트로 "AI 분석 중 (Gemini API 호출)" / "로컬 폴백" 명확히 표시

## 7. 시간 기준 — 사용자 설정 항목으로

외부 문서가 고정값으로 박은 시간들 → **모두 사용자 설정** 으로:

| 외부 문서 고정값 | 설정 키 (예정) | 디폴트 |
|---|---|---|
| 아침 알람 06:30 | `settings.morningSlotTime` | "06:30" |
| 저녁 알람 22:00 | `settings.eveningSlotTime` | "22:00" |
| 18:00 이후 = 다음날 모드 | `settings.eveningCutoffHour` | 18 |
| 자동 잠금 15분 | `settings.autoLockMinutes` ✅ 이미 구현됨 | 15 |
| 3일 공백 = Recovery Mode | `settings.recoveryThresholdDays` | 3 |

`config/encryptionPolicy.js` 의 `spiritualLockSettings` 컬렉션이 이미 이런 필드들을 평문으로 정의해놨음. UI 만 만들면 됨.

## 8. 마이그레이션 전략

외부 문서는 깨끗한 신규 사용자만 가정. 실제로는 이미 데이터 누적된 상태에서 스키마 진화 중.

**현재까지의 처리**:
- 옵션 A (`f8273bb`, 2026-05-12) — `readDocument` 가 v1 평문도 양방향 호환. 즉시 데이터 복구
- 옵션 B (자동 마이그레이션) — 미구현. 시한폭탄은 아니지만 스토리지 정리 차원

**향후 원칙**:
- 스키마 변경 시 **reader 가 양방향 호환부터** (옵션 A 패턴)
- 자동 마이그레이션은 **idempotent + 진행률 UI + 부분 재시도** 가 갖춰진 후
- 사용자 신뢰가 최우선 — 백업 다운로드 후 강제 삭제 같은 옵션 C 는 비추

## 9. 데이터 손실 절벽

외부 문서: "마스터 비밀번호 + 24단어 둘 다 잃으면 복구 불가."

**현재**: 그대로. 24단어 복구 코드만 백업.

**검토 중 옵션** (아직 구현 X):
- 신뢰할 사람 1명 위임 키 (Shamir's Secret Sharing 2-of-3)
- 시간차 복구 (분실 신고 후 N일 대기 → 이메일 인증 → 부분 데이터 복구)
- 종이 백업 가이드 (24단어를 인쇄해 안전 보관)

당장은 "복구 코드를 인쇄해서 안전한 곳에 두세요" 가이드 강화로 시작.
